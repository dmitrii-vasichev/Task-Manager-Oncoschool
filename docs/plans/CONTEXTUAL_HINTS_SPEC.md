# Спецификация контекстных подсказок (этап 1)

Дата: 2026-02-23
Проект: Онкошкола Task Manager
Статус: Draft v1

## 1. Рамки этапа

- Формат помощи: только контекстные подсказки в интерфейсе.
- Отдельная вкладка/страница Help Center: вне scope этапа 1.
- Покрытие: все текущие web-модули и все роли (`admin`, `moderator`, `member`).
- Источник истины: фактический код frontend/backend, а не только README.

## 2. Карта текущего функционала (по коду)

## 2.1 Frontend маршруты

| Маршрут | Назначение | Ключевые сценарии | Ключевые ограничения |
| --- | --- | --- | --- |
| `/login` | Вход через Telegram WebApp / web-login / debug dev-login | Инициация входа, polling статуса, авто-логин через Telegram initData | Без токена доступ к приложению закрыт |
| `/` (Dashboard) | Оперативный обзор задач и встреч | Переключение `Мои/Отдел`, выбор отдела, быстрый переход в задачи/встречи | Отделы и скоуп ограничены правами видимости |
| `/tasks` | Kanban задач | Фильтры, drag-and-drop статусов, создание задачи | Изменение статуса и часть действий ограничены правами |
| `/tasks/[id]` | Детальная карточка задачи | Статусы, назначение, приоритет, дедлайн, описание, чек-лист, апдейты, напоминание | Матрица прав по роли/авторству/исполнителю |
| `/meetings` | Список встреч | Вкладки предстоящих/прошедших, поиск, создание встреч (для модераторов) | Создание и удаление встреч — только moderator/admin |
| `/meetings/[id]` | Детали встречи | Транскрипция, AI summary, задачи встречи, заметки, участники, расписание | AI/транскрипт и большая часть редактирования — только moderator/admin |
| `/team` | Команда и структура отделов | Поиск, дерево отделов, карточки, создание/редактирование участников, отделы | Управление ограничено moderator/admin; часть полей только admin |
| `/analytics` | Аналитика задач/встреч | Срез по отделу, KPI, графики, таблицы, инсайты | Доступ к данным зависит от department visibility и роли |
| `/broadcasts` | Telegram-рассылки | Выбор групп, форматирование HTML, планирование/мгновенная отправка, история | Только moderator/admin; ограничения Telegram по длине/медиа |
| `/settings` | Настройки уведомлений/AI/напоминаний | Подписки, target-группы, AI-провайдер, командные дайджесты | Страница под `ModeratorGuard`; часть блоков только admin |

## 3. Ролевая модель для подсказок

## 3.1 Роли и особенности

| Роль | Где основной риск непонимания | Что важно объяснять подсказками |
| --- | --- | --- |
| `member` | Почему не видит часть задач/действий | Область видимости, недоступные действия, как эскалировать |
| `member` (head/extra access) | Переключение отделов и фильтров | Что значит расширенный доступ к отделам |
| `moderator` | Операции с последствиями для команды | Назначения, массовые действия, расписания, рассылки |
| `admin` | Админские настройки и риски влияния | AI provider/model, Telegram targets, role/test changes |

## 3.2 Базовые правила показа подсказок

- Подсказка должна быть привязана к конкретному UI-контексту, а не к странице в целом.
- Для скрытых по роли действий показывать не кнопку, а мягкую подсказку причины ограничения.
- Для потенциально опасных действий показывать подсказку до подтверждения.
- Подсказки по времени всегда поясняют часовую зону (в проекте ключевая зона — МСК).

## 4. UX-модель контекстной помощи

## 4.1 Типы подсказок

- `Orientation`: где я и что здесь делать (первое посещение экрана).
- `Permission`: почему действие недоступно для текущей роли.
- `Validation`: что не так в текущем вводе и как исправить.
- `Action-impact`: что произойдет после подтверждения.
- `State`: почему пусто/нет данных в блоке.
- `Cross-link`: куда перейти для завершения сценария.

## 4.2 Форматы

- Короткий inline-helper под контролом (приоритет по умолчанию).
- Compact callout внутри карточки/блока (для state/action-impact).
- Tooltip для иконок и переключателей.
- Confirm-dialog hint для необратимых действий.

## 4.3 Правила текста

- 1-2 коротких предложения.
- Структура: `почему` + `что делать дальше`.
- Без общих фраз типа "обратитесь к документации"; только конкретное действие в системе.
- Термины и названия должны совпадать с UI-лейблами.

## 5. Карта контекстных подсказок по модулям

## 5.1 Dashboard (`/`)

Назначение: быстрый срез задач/встреч с учетом scope и роли.
Критичные сценарии: выбор отдела, переключение `Мои/Отдел`, интерпретация блоков "Просрочено/Не обновлялись".
Ограничения: scope зависит от доступных отделов и роли.

Рекомендуемые точки подсказок:

| Триггер | Подсказка | Роль |
| --- | --- | --- |
| Первый вход на Dashboard | "Переключатель `Мои/Отдел` меняет источник задач и встреч в блоках ниже." | Все |
| Нет доступных отделов | "У вас нет доступного отдела, поэтому показываются только личные задачи." | Member |
| Есть просроченные | "Просроченные задачи требуют приоритета: откройте карточку и обновите статус/дедлайн." | Все |
| Есть stale-задачи | "Задача без апдейта >3 дней попадает в этот блок. Добавьте обновление в карточке задачи." | Все |
| Блок `Ожидают назначения` | "Эти задачи без исполнителя. Назначьте ответственного в карточке задачи." | Moderator/Admin |

## 5.2 Tasks Board (`/tasks`)

Назначение: рабочий Kanban.
Критичные сценарии: фильтрация, drag-and-drop статусов, создание новой задачи.
Ограничения: не все пользователи могут менять статус/назначать.

Рекомендуемые точки подсказок:

| Триггер | Подсказка | Роль |
| --- | --- | --- |
| Фильтр отдела недоступен | "Фильтр отдела скрыт, потому что у вас только один доступный отдел." | Member |
| DnD отклонен по правам | "Вы можете менять статус только своих задач или задач, где вы исполнитель." | Member |
| Открытие создания задачи | "Если вы не модератор, задача будет создана на вас." | Member |
| Выбор `Не назначен` в фильтре | "Показываются задачи без исполнителя, их можно переназначить в карточке." | Moderator/Admin |

## 5.3 Task Detail (`/tasks/[id]`)

Назначение: полный lifecycle задачи.
Критичные сценарии: статусные переходы, назначение, чек-лист, апдейты, напоминание.
Ограничения: field-level permissions; reminder требует исполнителя и будущего времени.

Рекомендуемые точки подсказок:

| Триггер | Подсказка | Роль |
| --- | --- | --- |
| Переход статуса | "Доступные переходы статуса зависят от текущего статуса задачи." | Все |
| Блок напоминания без исполнителя | "Напоминание доступно только для задач с назначенным исполнителем." | Все |
| Сохранение напоминания | "Время задаётся в МСК и должно быть в будущем." | Все |
| Редактирование описания/дедлайна/приоритета | "Часть полей доступна автору задачи и модератору." | Member |
| Удаление задачи | "Удаление без восстановления. Убедитесь, что в задаче нет критичных договорённостей." | Moderator/Admin |

## 5.4 Meetings List (`/meetings`)

Назначение: контроль встреч.
Критичные сценарии: разделение upcoming/past, создание встречи, удаление с уведомлением.
Ограничения: создание/удаление встреч доступны moderator/admin.

Рекомендуемые точки подсказок:

| Триггер | Подсказка | Роль |
| --- | --- | --- |
| Переключение вкладок | "`Предстоящие` для оперативной работы, `Прошедшие` для поиска истории и summary." | Все |
| Пустой upcoming | "Создайте встречу или проверьте фильтр scope в URL (`scope=my`)." | Все |
| Кнопка `Новая встреча` | "Создание встречи сразу задаёт основу для задач и напоминаний." | Moderator/Admin |
| Удаление встречи | "Можно удалить с уведомлением участников в Telegram-группы." | Moderator/Admin |

## 5.5 Meeting Detail (`/meetings/[id]`)

Назначение: единое рабочее место встречи (транскрипт, summary, задачи, заметки).
Критичные сценарии: получение транскрипта, AI-парсинг, применение summary.
Ограничения: AI-операции требуют transcript и роль moderator/admin.

Рекомендуемые точки подсказок:

| Триггер | Подсказка | Роль |
| --- | --- | --- |
| Нет transcript, есть Zoom meeting id | "После завершения встречи транскрипт подтянется из Zoom автоматически; можно запросить вручную." | Moderator/Admin |
| Кнопка AI parse | "AI-парсинг использует текущие настройки провайдера из `Settings`." | Moderator/Admin |
| Применение summary | "После применения будут созданы задачи и уведомления по встрече." | Moderator/Admin |
| Tab `Tasks` пуст | "Обработайте транскрипцию через AI, чтобы автоматически создать задачи из встречи." | Moderator/Admin |
| Изменение участников | "Список участников влияет на упоминания и уведомления по расписанию." | Moderator/Admin |

## 5.6 Team (`/team`)

Назначение: структура команды и доступов.
Критичные сценарии: создание/редактирование участника, деактивация, отделы.
Ограничения: role/test operations partially admin-only.

Рекомендуемые точки подсказок:

| Триггер | Подсказка | Роль |
| --- | --- | --- |
| Создание участника | "`Варианты имени` нужны для более точного AI-матчинга в summary." | Moderator/Admin |
| Изменение роли недоступно | "Менять роли может только администратор." | Moderator |
| Деактивация участника | "Перед деактивацией выберите, что делать с незавершенными задачами: снять исполнителя или переназначить." | Moderator/Admin |
| Флаг `Тестовый` | "Тестовые участники скрыты из обычной статистики и общих счетчиков." | Admin |
| Удаление отдела | "Отдел нельзя удалить, пока в нём есть участники." | Moderator/Admin |

## 5.7 Analytics (`/analytics`)

Назначение: KPI и диагностические срезы.
Критичные сценарии: корректный выбор отдела, интерпретация метрик.
Ограничения: часть аналитики (members) доступна только moderator/admin.

Рекомендуемые точки подсказок:

| Триггер | Подсказка | Роль |
| --- | --- | --- |
| Смена отдела | "Срез аналитики зависит от выбранного отдела и ваших прав видимости." | Все |
| Пустой блок отделов | "Нет данных для выбранного скоупа или у вас ограниченный доступ к отделам." | Все |
| Блок `Просроченные` > 0 | "Используйте этот индикатор как сигнал для ручной приоритизации и перераспределения." | Moderator/Admin |

## 5.8 Broadcasts (`/broadcasts`)

Назначение: массовые Telegram-рассылки.
Критичные сценарии: выбор целей, HTML-формат, media limits, планирование.
Ограничения: Telegram caption limit 1024 с изображением, 4096 без изображения.

Рекомендуемые точки подсказок:

| Триггер | Подсказка | Роль |
| --- | --- | --- |
| Выбрана картинка | "С изображением Telegram ограничивает подпись 1024 символами." | Moderator/Admin |
| История статуса `failed` | "Откройте ошибку и проверьте корректность цели/формат HTML/длину сообщения." | Moderator/Admin |
| Планирование времени | "Отправка планируется в МСК, проверьте дату и время перед подтверждением." | Moderator/Admin |
| Нет Telegram-групп | "Добавьте цели в `Settings -> Telegram-группы` (admin)." | Moderator/Admin |

## 5.9 Settings (`/settings`)

Назначение: системные настройки уведомлений, AI, напоминаний.
Критичные сценарии: изменение AI, массовые reminder-операции, weekly digest.
Ограничения: страница только для moderator+, часть секций только admin.

Рекомендуемые точки подсказок:

| Триггер | Подсказка | Роль |
| --- | --- | --- |
| AI provider/model | "Смена модели влияет на качество и стоимость парсинга summary." | Admin |
| Bulk reminders | "Массовое включение применится ко всем активным участникам." | Moderator/Admin |
| Telegram targets editing | "Создание/редактирование групп доступно только администратору." | Moderator |
| Weekly digest enabled без targets | "Для включения weekly digest выберите минимум одну Telegram-группу." | Moderator/Admin |

## 6. Приоритизация внедрения

## 6.1 P0 (сразу)

- Meeting Detail: transcript -> AI -> tasks flow.
- Broadcasts: лимиты Telegram и последствия send/cancel.

## 6.2 P1

- Meetings list (управление upcoming/past + notify delete).
- Settings (AI/targets/reminders).
- Analytics интерпретационные подсказки.

## 6.3 P2

- Тонкие UX-подсказки для пустых/редких состояний.
- A/B-тестирование формулировок.
- Расширение на Telegram-бот (следующий этап, отдельно).

## 7. Roadmap внедрения (2–4 недели)

## Неделя 1

- Собрать `Hint Inventory` по всем текущим экранам (таблица `screen + trigger + role + copy + priority`).
- Утвердить единый шаблон текста и severity уровни.
- Зафиксировать baseline-метрики: drop-off, частые ошибки, частые forbidden/action failures.

## Неделя 2

- Реализовать инфраструктуру показа подсказок на frontend:
  - реестр подсказок;
  - резолвер по контексту (`route`, `role`, `state`);
  - локальное скрытие/дисмиссы.
- Включить P0 подсказки в `Dashboard`, `Tasks`, `Task Detail`, `Meeting Detail`.

## Неделя 3

- Подключить `Team`, `Broadcasts`, `Settings`, `Analytics`, `Meetings`.
- Добавить продуктовые события (`hint_shown`, `hint_dismissed`, `hint_action_clicked`).
- Пройти QA по ролям (`member`, `member-head`, `moderator`, `admin`).

## Неделя 4 (если нужен полный цикл)

- Проанализировать метрики, удалить шумные/дублирующие подсказки.
- Доработать тексты по данным usage.
- Подготовить v2 backlog (включая Telegram-бот).

## 8. Как анализировать проект дальше (операционный цикл)

- После каждого feature/change:
  - обновить `Hint Inventory`;
  - проверить новые ограничения ролей и новых edge cases;
  - добавить/скорректировать подсказки для новых пустых и error-state.
- Раз в спринт:
  - смотреть top 10 ошибок по API (`403`, `400`, `409`);
  - связывать их с отсутствующими/неэффективными подсказками.
- Definition of Done для любой новой фичи:
  - есть минимум 1 contextual hint для первого входа;
  - есть hint для permission-denied или empty-state;
  - есть hint для необратимых действий.

## 9. Критерии успешности

- Снижение `403/400` пользовательских ошибок в ключевых сценариях на 20-30%.
- Рост завершения целевых сценариев (создание задачи, применение summary, отправка рассылки) на 10-15%.
- Снижение запросов в ручную поддержку по базовой навигации/правам.
- Не более 15% подсказок массово dismiss без взаимодействия (сигнал "шум").

## 10. Итог этапа 1

- Контекстная помощь внедряется как слой в текущие экраны.
- Отдельная вкладка "Справка" не создается на этом этапе.
- Приоритет отдан зонам с максимальным количеством role/state ошибок.
